# 06 · 新人上手指南（零基础也能用）

面向**刚加入、没怎么写过程序、第一次接触 MR 的新人**。

读完这一篇你就能提交第一个合规的 MR。不需要懂 CI、不需要记规范细节——那些都由 AI 和自动化脚本兜底。

---

## 先理解一件事：你不是来"填表"的

这套治理系统的设计目标，就是让你**几乎不用学**。真正干活的是 AI：你说需求，AI 写代码、跑测试、填模板、提交 MR。

> **你的角色 = 提需求 + 拍板。不是填表。**

下面分两种情况：用 AI 工具（推荐）、和手动填写（备用）。

---

# 场景 A：用 AI 工具开发（最省事）

如果你用 Claude Code / Cursor / 通义灵码 等 AI 工具，流程是这样的：

## 第 1 步：用大白话告诉 AI 你要什么

像聊天一样说清楚，**不需要任何术语**：

```
帮我做个功能：用户在投放任务里重复点提交时，不要重复建任务，要去重。
```

把"想要什么"说清楚就行。说得越具体，AI 做得越准。

## 第 2 步：让 AI 走完整提交流程

AI 写完代码后，对它说：

```
请按仓库规范完成提交：跑测试、加必要的风险注解、提交，并自动创建 MR。
```

AI 会自动做完这些（你不用操心细节）：

| AI 自动做的事 | 对应的治理要求 |
|---|---|
| 记录用了多少 AI 写代码 | AI-Usage 自动采集 |
| 跑单元测试并留痕 | 测试痕迹检测 |
| 扫描有没有风险代码、该加注解的加上 | 风险注解硬门禁 |
| 提交，自动盖 commit 标记 | commit trailer |
| 调 `create_mr.py` 自动生成 MR | MR 描述自动生成 |

## 第 3 步：检查 AI 生成的 MR，然后合并

AI 会给你一个 MR 链接。打开它，确认两件事：

- **背景**写得对不对（是不是你要的功能）
- **变更内容**大致对不对

没问题就点"合并"（或让 AI 帮你合）。**全程你没填过任何表格。**

> 装了 DeliverHQ 的项目更省事：连"背景"都不用 AI 写——系统会按分支名里的需求编号（如 `CR-1234`）自动从需求文档里读。

---

# 场景 B：手动填模板（不用 AI，或想看懂模板）

如果你要自己在 GitLab 网页上填 MR，照下面做。

## 怎么打开模板

1. 在 GitLab 仓库页面，把你的分支发起一个 **Merge Request**
2. 描述框会**自动加载模板**（已经帮你列好段落标题）
3. 照着填下面几项

## 各段落怎么填（关键：用大白话）

### `## 背景`——为什么做这个改动

写**问题或需求**，不是写"改了代码"。

| ✅ 好 | ❌ 不好 |
|---|---|
| `用户重复点提交会建重复任务，需要去重` | `修改了代码` |
| `运营要求支持微信支付，目前只有支付宝` | `加功能` |

### `## 变更内容`——你改了啥

列 2-5 条要点：

```
- 提交前检查任务是否已存在
- 重复时给用户提示"任务已创建"
- 加了对应的单元测试
```

### `## 自测确认`——你怎么验证它能用

写**测了什么、看到什么**，别只写"测过了"。

| ✅ 好 | ❌ 不好 |
|---|---|
| `- [x] 本地连点两次提交，第二次被拦住` | `- [x] 已测试` |
| `- [x] 跑了单元测试，12 个全过` | `- [x] OK` |

### `## 风险与回滚`——万一出问题怎么办

小改动直接写：

```
低风险，无需特别说明
```

大改动（改了很多文件、动了部署配置或数据库）才需要详细写风险点和怎么回退。

### 其他段落

`## 不包含的内容`、`## 关联` 是可选的，没有就写"无"或留空。

## 不用你管的部分

MR 描述底部有个 **"📊 治理元数据"** 折叠块（AI 使用情况、测试结果等），那是系统**自动生成**的，**你不用碰也不要删**。

---

# 新人最容易踩的 3 个坑

### 坑 1：背景写成"改了什么"

背景要回答**"为什么"**。「修复 bug」太空，「用户重复提交会建重复任务，需去重」才对。

### 坑 2：自测确认只写"测过了"

要写**具体测了什么**。「点两次提交，第二次被拦」这种有信息量的才有用。

### 坑 3：CI 报红就慌

如果合并前系统提示红色错误（比如"有风险代码没加说明"或"没有测试记录"），**别慌也别硬合**：

- 用 AI：把错误提示原样发给 AI，说"CI 报这个错，帮我修"，它会处理。
- 不用 AI：看错误里的 `fix:` 提示，照着做；看不懂就问团队里的技术负责人。

常见提示和含义见 `05-ci-reference.md` 的"排查速查表"。

# 常见疑问

**Q：我完全不懂代码，能保证 MR 合规吗？**
能。风险扫描、测试检查、AI 标记都是机器自动做的。你只要把"为什么改、改了啥、怎么验的"用大白话说清楚。

**Q：AI 帮我提交，出了问题算谁的？**
你是这个 MR 的提交人，要对合并负责——所以第 3 步的"看一眼确认"很重要。看不懂的地方就问 AI"这个改动有什么风险"，或找技术负责人。

**Q：CI 一直红，AI 也修不好怎么办？**
别反复硬试。把错误贴给技术负责人，或在 `05-ci-reference.md` 查排查表。

**Q：什么是"风险注解"？我需要自己写吗？**
是给"看起来可疑但其实合理"的代码加的说明（比如某个账号故意免登录）。**正常你不用手写**——AI 会在扫描到时自动加。详见 `../risk-types.md`，但新人不用先读。

---

# 一句话总结

| 你的情况 | 你要做的 |
|---|---|
| **用 AI** | 说需求 → 让 AI 跑完流程 → 看一眼 MR → 合并 |
| **不用 AI** | 填 4 个段落，大白话回答"为什么改、改了啥、怎么测的"，剩下机器自动补 |

剩下的——风险扫描、测试检查、AI 标记、合规判断——**全是机器在做，不用你记**。
